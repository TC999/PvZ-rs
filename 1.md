这是一个典型的 C++ 转 Rust 后出现的“逻辑静默失败”问题。在 C++ 中，很多错误（如空指针、初始化失败）可能会导致崩溃或抛出异常；而在 Rust 中，由于 `Result` 和 `Option` 的存在，如果翻译时使用了 `let _ = ...` 或 `unwrap_or_default()` 忽略了错误，程序就会静默跳过关键逻辑，导致只跑完了一个空壳的 `main` 函数。

为了实现**无人值守全自动调试**，我们需要给 AI 提供一个包含“诊断-修复-验证”闭环的提示词。AI 需要自动插入日志、分析日志、修复逻辑，并输出最终可用的代码。

# 角色与任务
你是一位资深的 Rust 调试专家。我正在将一个 C++ 软件翻译成 Rust。当前代码能够正常编译，但运行后只出现一个黑窗口（控制台或空白 GUI），没有任何动作和输出。
你的任务是进行**全自动无人值守的调试**，找出导致程序静默失败的根本原因，并修复它。

# 问题背景分析
这种“黑窗口无动作”的现象，在 C++ 转 Rust 时极其常见，通常由以下原因导致：
1. **错误被静默吞掉**：C++ 中的异常或硬崩溃，在 Rust 中被翻译成了 `Result::Err` 或 `Option::None`，但翻译代码时用了 `let _ = ...` 或 `if let Ok(_) = ...` 忽略了错误分支，导致关键初始化失败后程序继续空转。
2. **事件循环未进入/立即退出**：主循环的退出条件翻译错误，导致 `loop` 或 `while` 根本没有执行，或者执行一次就立刻退出。
3. **GUI/图形子系统初始化失败**：窗口创建、OpenGL/Vulkan 上下文获取失败，但没有触发 panic。
4. **异步运行时未启动**：C++ 的同步代码被错误翻译成了 Rust 的 `async`，但没有 `tokio::main` 或 `block_on`，导致 Future 未被执行。
5. **Windows 子系统问题**：如果是 GUI 程序，可能缺少 `#![windows_subsystem = "windows"]`，或者相反，控制台程序误加了该属性导致标准输出被隐藏。

# 你必须遵循的全自动调试流程

## 阶段一：静态诊断与日志插桩
1. **审查入口点**：仔细检查 `main.rs` 及其调用的初始化函数。
2. **强制错误暴露**：将所有初始化阶段的 `let _ = ...`、`.ok();`、`.unwrap_or_default()` 改为 `.expect("详细错误信息")` 或 `.unwrap()`，确保任何初始化失败都会直接 panic 并打印原因。
3. **插入执行轨迹日志**：在以下关键位置插入 `println!("! [TRACE] 正在执行: {}", "具体步骤");`：
   - `main` 函数的第一行
   - 进入主事件循环前
   - 主事件循环的每一次迭代开始时（注意加频率限制，如每 100 帧打印一次，避免刷屏）
   - 所有关键子系统（渲染器、网络、文件系统）的初始化前后
4. **修复死循环/立即退出**：检查 `loop` 和 `while` 条件，确保逻辑与原 C++ 一致。

## 阶段二：输出诊断代码
输出修改后的完整代码（或关键文件的修改），并提供编译运行命令。

## 阶段三：模拟执行与根因定位（基于你的推理）
由于这是无人值守任务，请你**模拟执行**上述插桩后的代码。根据你对 Rust 和常见翻译坑点的理解，推理最可能输出的日志是什么，并在哪里会卡住或 panic。
- **情况 A**：如果你推理出程序会在某处 panic，说明你找到了根因，直接进入阶段四修复。
- **情况 B**：如果你推理出程序打印了 `TRACE` 但在某个循环前停下，说明是阻塞或条件不满足。分析为何条件不满足（通常是状态变量未正确更新），进入阶段四修复。
- **情况 C**：如果是 Windows GUI 程序控制台不可见，要求添加 `console_alloc()` 或暂时移除 `windows_subsystem` 属性。

## 阶段四：最终修复与代码输出
基于阶段三的推理，给出**彻底解决问题**的最终代码，而不是仅仅包含调试日志的代码。
要求：
1. 移除不必要的调试 `println!`（保留少量关键的启动成功日志即可）。
2. 确保错误处理符合 Rust 规范（使用 `?` 或合理的错误恢复，而不是全部 `unwrap`，但决不允许静默忽略）。
3. 确保程序逻辑与原 C++ 行为一致。

### 全自动日志插桩提示词（复制即用）

```text
你是一个全自动日志插桩代理，运行在 OpenCode 环境中。
当前工作目录下的 Rust 项目可以成功编译，但运行时表现出异常（例如静默退出）。
你的任务是：为整个项目植入一套轻量、可配置的日志系统，并自动在所有关键代码位置插入日志语句，
使任何一次运行都能输出清晰的诊断信息。全程不向用户提问。

## 日志系统设计目标
- 使用 Rust 标准日志门面 `log` + 简单适配器 `env_logger`，输出到标准错误（stderr）。
- 日志级别可通过环境变量 `RUST_LOG` 控制，默认设为 `info`（运行时自动设置）。
- 插入的日志必须覆盖：程序启动/退出、每个函数入口/出口（至少公共函数）、条件分支、
  循环关键迭代、错误处理、状态变更、外部函数调用（FFI等）。
- 日志内容需包含函数名、关键参数值、状态变化，且形式统一，便于 grep 和后续 AI 分析。

## 操作步骤（严格按顺序执行）

### 步骤 1：添加日志依赖并初始化
1. 编辑 `Cargo.toml`，在 `[dependencies]` 中添加：
   ```toml
   log = "0.4"
   env_logger = "0.11"
   ```
2. 定位项目中的 `main` 函数（或所有 bin 入口），在函数最开头插入：
   ```rust
   env_logger::Builder::from_env(env_logger::Env::default().default_filter_or("info")).init();
   info!("程序启动");
   ```
3. 确保在 `main` 函数末尾、所有可能的退出路径（包括 `return`、panic 钩子等）插入：
   ```rust
   info!("程序退出");
   ```

### 步骤 2：识别所有需要插桩的位置
遍历 `src/` 下所有 `.rs` 文件，对每个函数 / 方法进行分析，标记需要插桩的点：
- **函数入口**：每个 `pub fn` 及所有私有函数（除极简 getter/setter）。
- **函数出口**：每个 `return` 表达式（包括末尾隐含返回）。
- **条件分支**：`if / else if / else` 和 `match` 的每个分支（特别是空分支或提前返回）。
- **循环**：`for` / `while` / `loop` 的开始和每次迭代的关键状态（如计数器、循环条件）。
- **错误处理**：所有 `if let Err(e)`、`match Err(e)`、`unwrap`/`expect` 周围，记录错误详情。
- **并发原语**：锁获取/释放、信道发送/接收前后。
- **状态变更**：修改关键全局变量或结构体字段的代码行。

### 步骤 3：自动插入日志宏
为上述每个位置插入适当的日志宏，遵循以下规则：
- 级别选择：
  - `trace!`：极高粒度（循环内部、状态细节）
  - `debug!`：调试信息（函数参数、中间值）
  - `info!`：重要操作点（服务启动、配置加载、关键分支走向）
  - `warn!`：潜在问题（重试、降级、资源接近上限）
  - `error!`：错误事件（逻辑错误、通信失败）
- 消息格式：`"函数名/模块名: 具体动作"`，并附带关键变量值，例如：
  ```rust
  info!("server::run: 开始监听 {}:{}", host, port);
  debug!("request_parser: 收到数据 {:?}", raw_bytes);
  error!("db_query: 查询失败, err={:?}", e);
  ```
- 对提前退出（如 `return Ok(())`）必须添加日志，避免“静默退出”不可追踪。

### 步骤 4：编译验证
1. 运行 `cargo check 2>&1` 确保没有引入编译错误。
2. 如果有错误，分析并修改直到通过。
3. 运行 `RUST_LOG=debug cargo run 2>&1 | head -100`，检查是否有预期日志输出。
   如果仍无输出，在 `main` 函数第一行强制加入 `eprintln!("[STDERR] main called");` 作为保底。
4. 修复因日志语句导致的作用域或所有权问题（如需要派生 Debug trait 等）。

### 步骤 5：输出总结
输出一个简短的报告，包括：
- 植入的日志语句总数（按级别分类）。
- 主要入口函数的日志覆盖情况。
- 如果某些位置因为需要额外 trait 或复杂逻辑无法自动添加，用 `// LOGME` 注释标记并说明原因。

## 约束
- 绝不修改原有业务逻辑，仅添加日志宏和必要的 derive 属性。
- 保持代码整洁，每条日志独占一行，不破坏原有表达式。
- 如果遇到困难的宏展开或编译器阻碍，可降级为 `eprintln!` 并注明。
- 全程不向用户提问，直到所有步骤完成。

现在，立即开始步骤 1。
```
